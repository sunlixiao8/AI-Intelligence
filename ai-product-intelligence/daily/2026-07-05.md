### 1. Churchcalls AI

- **产品详情**
  - **名称：** Churchcalls AI
  - **类别：** 人工智能 / 垂直行业 SaaS
  - **目标用户：** 教会管理人员、牧师、宗教组织行政人员
  - **URL：** https://trustmrr.com/startup/churchcalls-ai (来源: TrustMRR)

- **用户问题**
  - **痛苦问题：** 教会通常缺乏专职的 IT 和行政人员，需要处理大量的重复性沟通工作，如电话、短信提醒、活动通知、探访预约等。这占用了牧师和志愿者的宝贵时间，影响了核心的牧养和社区建设。
  - **付费理由：** 这是一个工作问题（而非生活乐趣），有明确的预算（教会奉献款或行政经费）。用户花钱是为了“省时”和“提高社区参与度”，不是为了“好玩”。

- **商业信号**
  - **定价模式：** 未知，根据售价推测可能提供月度订阅 ($10-$50/月)。(低置信度)
  - **月收入估算：** **$210/月**
  - **置信度：** **高** (来源: TrustMRR, 直接列出当前 MRR 为 $210。证据链接同上。)
  - **增长/牵引力证据：**
    - 在 TrustMRR 上挂牌出售，定价 $50k，倍数高达 19.8x。高倍数通常意味着产品或客户群具有高粘性和长期价值，尽管当前收入较低。
    - 存在感强烈，表明它抓住了“宗教领域”这个细分市场。

- **为什么可能赚钱**
  - **使用频率：** 高。每周都有活动通知、祈祷会、礼拜安排等。沟通是教会的生命线。
  - **紧迫程度：** 中-高。教会社区参与度下降是许多教会的痛点，自动化的沟通工具能直接提升参与度。
  - **预算负责人：** 教会行政主任或牧师，预算较灵活，且对提升效率的工具付费意愿明确。
  - **切换成本：** 一旦教会将成员名单、联系方式、活动日历集成进去，切换成本会很高。

- **产品逻辑与技术路径**
  - **适合初学者的 MVP 范围：** 一个针对单个教会的“通知与回访”系统。MVP 提供：1) 电话拨号通知；2) 自动发送短信/邮件；3) 一键呼叫（用于上门探访）。
  - **核心工作流：**
    1.  管理员上传成员名单（CSV）。
    2.  创建活动（如“下周二小组聚会”）。
    3.  系统自动拨打电话或发送短信进行通知。
    4.  成员可按键回复（1=参加, 2=不参加, 3=需要交通帮助），系统自动记录并提醒管理员。
  - **需要的数据或输入：** 教会成员名单（姓名、电话、偏好联系方式），活动日历，预设的语音或短信模板。
  - **AI 模型用法：**
    - **语音生成 (TTS)：** 使用 OpenAI TTS 或 ElevenLabs 生成不同角色的语音（牧师、同工）。
    - **意图识别 (NLP)：** 使用 GPT-4o-mini 或 Claude Haiku 分析用户回复的语音或按键反馈。
    - **智能体：** 用简单的 LLM Call 来根据回复更新成员状态，并决定下一步动作（如给交通组发通知）。
  - **推荐技术栈：**
    - **语音 API：** Twilio（权威推荐）用于电话拨打和 SMS。
    - **语音 AI：** 使用 Deepgram 或 AssemblyAI 实现语音识别（STT），用于分析留言。
    - **后端/数据库：** Supabase（管理成员、活动、响应记录）。
    - **前端：** Next.js + Tailwind CSS。
    - **AI 编排：** 简单的 Python FastAPI 或 Supabase Edge Functions（用于处理语音转文本和逻辑判断，避免复杂的 Agent 框架）。
  - **最大难点：**
    - **电话与语音交互的可靠性：** 真实电话场景中，背景噪音、方言、不清晰的语音会给 STT 和意图识别带来巨大挑战，错误率会很高。
    - **教会领域知识的收集：** 需要深度理解教会的内部流程（如新人跟进流程、受洗课程通知等），才能将自动化做得“像人一样”。
  - **降低难度的捷径：**
    - 第一版可以不做电话，只做 **短信/邮件自动化**，这是最可控的方案。使用 Twilio SMS 和 SendGrid。
    - 不做复杂的语音对话树。使用 **“预录语音” + “按键菜单”**，完全绕开 AI 语音识别，只保留最核心的自动化。

- **适合 OPC 初学者的机会**
  - **小众版本：** 针对 **“在线教会”** 或 **“青年团契”** 的更轻量版工具，专注于 Discord/Slack 社群的自动化事件提醒和签到。
  - **第一批客户：** 去你所在城市或线上的多家小型教会，访谈行政助理或牧师，了解他们目前如何管理通知。询问：“你每周在发通知和跟进这件事上花费多少小时？”
  - **1 周验证实验：** 手动操作：用谷歌 Calendar + 邮件合并，为一周内的活动替一个教会手动发送 3 次定制短信/邮件。问牧师：“如果有一个软件能帮你每周自动完成这个工作，每月只需 $30，你会愿意付费吗？”

    1.  用户输入文本 prompt 或上传2D图片。
    2.  模型（如 TripoSR）在数十秒内生成一个粗糙的 3D 网格。
    3.  提供自动拓扑优化、纹理贴图、动画等功能。
  - **需要的数据或输入：** 文本描述或图片。需要海量 3D 数据对进行训练（如 Objaverse、ShapeNet）。

  - **推荐技术栈：** 这个产品需要自己训练模型，技术栈涉及 PyTorch / TensorFlow、NVIDIA GPU 服务器集群、CUDA、3D 处理库等。对个人开发者而言，核心技术栈过高。
  - **最大难点：**
    - **训练数据与算力：** 需要处理海量的 3D 数据和巨大的GPU算力开销（动辄数十万甚至百万美元）。
    - **生成质量与一致性：** 解决 3D 模型的几何细节、纹理清晰度、与文本/图片一致性是学术和工业界的共同难题。
    - **实时性与可控性：** 用户期望快速生成并能精细控制细节（如修改某个部件），这对模型架构有极高要求。
  - **降低难度的捷径：**
    - **放弃自己训练，使用 API：** 集成 Meshy、Tripo AI 或 OpenAI 的 3D 生成 API（如果开放）。将自己定位为 **“AI 3D 模型管理 / 应用平台”**。比如：
        - **AI 游戏资产批处理工具：** 用户上传垃圾模型（如用 Tripo 生成），你的工具自动调整尺寸、法线、压缩、格式转换（FBX/GLTF），并一键上传到 Unity/Unreal 商店。
        - **AI 电商 3D 展示生成工具：** 用户上传商品图片，你的工具调用 3D API 生成模型，然后自动生成 360 度旋转视频或 AR 预览代码。

  - **小众版本：** **“AI 图生3D角色卡”**：为桌游（TRPG）、DnD玩家群体服务。用户上传一张角色设定图，你的工具将其快速处理成一个适合打印的棋盘游戏微缩模型文件（STL），或用于 VTT 平台（如 Roll20）的 3D 角色。

## 今日模式

**模式：在「高频率、高压迫性」的垂直领域做减法，比在广阔市场做加法更容易成功。**

- **Churchcalls AI** 在宗教沟通（高频率，对出勤和关怀有极高要求）领域做减法（仅用 AI 电话通知）。
- **Sadece Islam** 在宗教仪式（高频次、高精准、高精神压迫）领域做减法（只做祈祷时间记录）。
- **Tripo AI** 虽然在广阔市场，但其核心逻辑是“将复杂的 3D 建模（高痛苦、高成本）用 AI 做简单（一键生成）”。对于个人开发者，只应做**外层的“简单”**，不要试图挑战**内层的“复杂”**（核心模型）。

**经验：** 不要试图去教育用户“你需要这个工具”。而是找到一群“不做这事就会很痛苦”的用户，然后用最简单的方式（甚至不需要 AI）去解决他们最核心的 1-2 个痛苦。

## 建造雷达

1.  **为在线社区 KOL 服务的自动语音互动系统：** 类似于 Churchcalls，专为 YouTube 主播、独立创作者设计。当社区有新成员或直播开始时，自动用主播的声音给核心粉丝打电话/发语音消息通知（例如：“xx来啦！直播开始啦！想你了！”）。痛点：维护深度粉丝关系。
2.  **特定运动类型的训练动作 3D 生成器：** 调用 Tripo 等 API。用户上传一张高尔夫/网球/羽毛球挥拍动作的截图，AI 自动生成该动作的 3D 动画模型，用于分析。痛点：体育老师或教练需要大量的教学动画。
3.  **跨平台的祈祷/冥想/记录同步 App：** 使用最基础的 Adhan 库做后端，专注于 **“更美观的 UI”** 和 **“更简单的数据导出”**。用户可以在 Apple Watch 上快速记录一次祈祷，数据同步到 Android 平板。不收费，全靠广告（但广告要优雅），赚取活跃用户。

## 用户访谈问题

1.  **（对职场/行政人员）** “在您目前的工作流程中，最让你觉得‘浪费生命’的重复性沟通工作是什么？（例如：电话通知、群发邮件、数据录入、审批提醒）你认为把它自动化需要每月投资多少钱你才愿意尝试？”
2.  **（对 DnD 玩家 / 3D 爱好者）** “当你和别人描述你梦想中的角色时，最让你觉得沮丧的是‘自己画不出’还是‘别人理解不了’？” (用于验证需求的类型，是生成工具还是交流工具)。
3.  **（对宗教 App 重度用户）** “你使用一款 App 进行祈祷/冥想/记录的最高频场景是什么？（例如：换季时的“午夜祈祷提醒”？海外旅行时的“清真餐厅地图”？）如果你只能保留一个功能，那会是哪个？”
